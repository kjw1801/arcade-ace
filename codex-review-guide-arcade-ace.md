# codex-review-guide.md

> 《국정원 에이스는 오락실 관리자를 선배님이라 부른다》 회차 cross-review 운영 가이드.
> 원칙: **루브릭을 주고 / 플래그로만 받고 / 작가가 판단하고 / 붙여넣기로 효용부터 측정.**
> 지금 단계는 CLI/MCP 연동 없이 ChatGPT(GPT-5) 붙여넣기로 운영한다.

---

## 1. 운영 흐름 (회차당)

1. Claude Code에서 회차 집필 → `/polish` 까지 완료.
2. ChatGPT 새 대화에 **[리뷰 프롬프트](#2-리뷰-프롬프트-붙여넣기용)** + 회차 원고 붙여넣기.
3. GPT가 **플래그 리스트**로 응답 (수정 금지, 지적만).
4. 정우(작가)가 각 플래그를 **채택 / 기각 / 보류**로 분류.
5. 채택한 플래그만 Claude Code로 가져가 수정 요청 (`아래 플래그만 반영해줘: …`).
6. 회차 효용 로그를 [측정표](#4-효용-측정-gono-go)에 기록.

> 핵심: GPT와 Claude를 직접 합의시키지 않는다. 충돌하면 **작가가 최종 결정**.

---

## 2. 리뷰 프롬프트 (붙여넣기용)

```
너는 노벨피아 웹소설 전문 리뷰어다. 아래 회차 원고를 평가한다.

[작품 정보]
- 제목: 국정원 에이스는 오락실 관리자를 선배님이라 부른다
- 장르: 노벨피아 / 현대 첩보 + 착각계 코미디
- 주인공 이정우: 본인만 자기가 평범하다고 믿는다. 진짜 능력이 첩보 스킬로
  오독되는 '착각' 구조가 핵심.

[반드시 지킬 규칙]
1. 원고를 다시 쓰지 마라. 문장을 새로 써주지 마라. '약한 지점'을 지적만 하고,
   고칠 방향은 한 줄 힌트로만 제시한다.
2. 작품 고유 설정의 옳고 그름은 판단하지 마라. (누가 누구를 어디까지 오해
   중인지, 설정 충돌 여부 등은 작가가 별도 관리한다. 모르는 설정을 틀렸다고
   단정하지 마라.)
3. 아래 [평가 축]으로만 평가한다. 축 밖의 일반론(취향성 코멘트)은 쓰지 마라.

[평가 축]
- HOOK1K: 첫 1,000자 안에 독자를 잡는 '갭 펀치'(평범↔괴물 오해의 강한 대비)가
  있는가? 도입이 늘어지는가?
- LIGHT: 라노벨 톤이 적정한가? (정우 시점 내적 독백은 짧은 츳코미체인가,
  무겁게 처지는 구간은 없는가, 코팅이 과하거나 부족한가)
- POV_BALANCE: 외부 인물 초점(스콜피온/강태오/정하린)이 '착각을 강화'하는
  펀치 역할을 하는가, 아니면 분량만 잡아먹는가? 동일 장면의 초점 이동에 불필요한
  구분선이 들어가지는 않았는가? 정우 밀착이 충분한가?
- MOBILE: 문단·문장 길이와 대사 비중이 노벨피아 모바일 가독성에 맞는가?
  (긴 서술 덩어리, 낮은 대사 비중 지적)
- TONE_CONSISTENCY: 인물별 말투가 흔들리는 문장이 있는가?
  (정우=당황한 평범체 / 정하린=깍듯한 오해체 / 외부=건조한 관찰체)

[출력 형식]
■ 한 줄 총평: (이 회차의 전반 인상 1문장)

■ 플래그 목록: (약한 지점만. 각 항목 아래 형식)
- [축] | 위치(문단/대사 인용 5단어 이내) | 문제(1문장) | 심각도(상/중/하) | 고칠 방향(한 줄)

■ 살릴 강점: (잘 된 지점 2개 이내, 각 1줄 — 수정 중 날리지 않도록)

■ TOP 3: 가장 먼저 손볼 플래그 3개를 심각도순으로.
```

> 위 프롬프트는 회차마다 그대로 재사용. 작품 정보 블록만 고정이고 원고만 교체한다.

## 3. 작가 판단 규칙 (플래그 처리)

받은 플래그를 다음 기준으로 분류한다.

- **채택**: 정우 님이 봐도 약한 게 맞다 → Claude Code 수정 목록에 넣는다.
- **기각**: 의도된 연출이거나 작품 색이다 (예: 외부 시점 컷을 일부러 길게 깐 복선).
  GPT는 design 문서를 모른다 — 기각에 죄책감 갖지 말 것.
- **보류**: 지금 회차에선 못 정하지만 패턴으로 누적되면 볼 것 → learnings.md에 메모.

> 충돌 처리: Claude polish는 살리자 했는데 GPT는 빼자 한다 → **작가가 결정**.
> 두 AI를 합의시키려 하지 말 것.

Claude Code 전달 예시:
```
EP001 아래 플래그만 반영해줘 (나머지는 의도된 거라 두자):
- HOOK1K: 도입 첫 문단이 늘어짐 → 정하린 대사로 시작하게 앞으로 당기기
- MOBILE: 3번째 문단 서술 덩어리 분할
```

## 4. 효용 측정 (Go/No-Go)

이 구조를 상시화할지 말지는 **"GPT가 Claude polish 못 잡은 걸 실제로 잡나"** 로 정한다.
EP001~003 동안 회차마다 기록:

| 회차 | GPT 플래그 총수 | 그중 채택 | '새로운' 채택* | 메모 |
|------|----------------|-----------|---------------|------|
| EP001 |               |           |               |      |
| EP002 |               |           |               |      |
| EP003 |               |           |               |      |

\* '새로운' = Claude polish가 이미 잡았던 것 말고, GPT만 새로 짚은 채택 건.

**판정 기준**
- 3회차 평균 '새로운 채택'이 회차당 **1건 이상** → 효용 있음. 계속 사용, 추후 CLI 자동화 검토.
- **0~1건 사이** → 미미. 가끔(중요 회차/1화·분기점)만 돌린다.
- 거의 **0건** → 중단. Claude polish로 충분. 시간 아낀다.

## 5. 나중에 CLI 연동으로 넘어가는 조건 (지금 아님)

아래가 충족되면 그때 OpenAI 공식 Codex 플러그인 + MCP 연동을 검토한다.
- 효용이 확인됐고(4번 Go),
- 회차가 수십 개 쌓여 '전체 회차 톤 대조' 같은 자동화가 필요해지고,
- 붙여넣기 반복이 실제 병목이 될 때.

그 전까지는 붙여넣기가 가장 빠르고 싸다.
